How the LEGO estimator works

This calculator is designed as a linked estimation model. The user can start from any available information: model dimensions, desired part count, desired weight, void percentage, or small/large detail balance. The calculator then automatically updates every unlocked value that depends on it.

The main idea is that all parameters describe the same LEGO model from different angles.

Dimensions define the outer bounding box of the model: length, width, and height. Together they create the envelope volume.

Void describes how much of that outer volume is empty space. For example, a model with 50% void means only half of the outer bounding volume is actually occupied by LEGO structure.

Detail balance describes the balance between large hollow structural parts and small dense decorative parts. A low value means the model is built mostly from large parts, so it is lighter and uses fewer pieces. A high value means more small parts, so the model becomes heavier and the part count increases.

Part count is estimated from the occupied volume and the detail balance. More small details increase the number of parts needed for the same model volume.

Weight is estimated from occupied volume, plastic density, and internal fill. Large LEGO structures are treated as more hollow, while small details are treated as denser.

Locking logic

Every major parameter has a lock. A locked value is treated as fixed. It will not be recalculated by the system.

However, a locked value can still be changed manually by the user. When the user changes a locked value, that new value becomes the new fixed target, and the rest of the unlocked parameters adapt around it.

The general rule is:

  • The value you move becomes the active driver.
  • Locked values stay fixed.
  • Unlocked values are recalculated automatically.
  • At least two parameters must remain unlocked, otherwise the model has no room to adapt.

This makes the calculator behave like a constraint system rather than a one-way form.

Scenario 1: only dimensions are changed

If the user changes length, width, or height, the calculator treats the new dimensions as the main geometric input.

The envelope volume is calculated from:

Length × Width × Height

Then the void percentage is applied to estimate the occupied LEGO volume. From that occupied volume, the calculator estimates the part count and weight.

This scenario is useful when the client gives approximate dimensions, or when the model size is already known from a Stud.io file.

Scenario 2: one dimension is fixed, the others adapt

If the user locks one side, for example length, and then changes the target weight or target part count, the calculator keeps that length fixed.

The other two dimensions are recalculated equally so the model keeps a balanced rectangular shape.

Example:

  • Length is locked.
  • Desired weight is changed.
  • Width and height update together.
  • Part count updates from the new volume and detail balance.

This is useful when the client has a strict maximum length, but the other dimensions can still change.

Scenario 3: two dimensions are fixed

If two dimensions are locked, the third dimension becomes the main adjustable side.

Example:

  • Length and width are locked.
  • Desired part count is changed.
  • Height changes to make the new volume possible.

This is useful for models that must fit a footprint, such as a display base, shelf, box insert, or packaging constraint.

Scenario 4: all three dimensions are fixed

If length, width, and height are locked, the physical model size is fixed.

In that case the calculator cannot change the volume. Instead, it recalculates the unlocked outputs: part count, weight, and effective structure.

This scenario is useful when the model dimensions are already final and the calculator is only used to estimate production impact.

Scenario 5: only desired part count is changed

If the user changes only the desired part count, the calculator estimates the volume needed to reach that number of parts.

It uses the current void percentage and detail balance as assumptions.

Then it creates a proportional model size from that estimated volume, usually as a cube unless some dimensions are locked.

This is useful at the planning stage when the project has a part-count target but the exact dimensions are not known yet.

Scenario 6: only desired weight is changed

If the user changes only the desired weight, the calculator estimates how much occupied LEGO volume is needed to reach that weight.

It uses the current detail balance to decide how dense the model structure is.

Then it adds the selected void percentage and converts the result into outer dimensions.

This is useful when shipping weight, packaging weight, or material cost is the main constraint.

Scenario 7: void is changed

Void controls how much empty space exists inside the bounding volume.

If void increases, the same occupied LEGO volume needs a larger outer bounding box. If the outer dimensions are locked, increasing void reduces the estimated occupied structure and therefore reduces weight and part count.

If void is locked, it becomes a fixed design assumption. Other values must adapt around it.

This is useful because different model types have different void behavior. A compact cube-like object has lower void. A tall architectural model, open structure, or landscape scene can have much higher void.

Scenario 8: detail balance is changed

Detail balance changes the relationship between large structural pieces and small dense details.

If detail balance increases, the same volume tends to require more parts and more plastic mass. If weight or part count is locked, the calculator compensates by changing the available unlocked values, usually dimensions or occupied volume.

This makes the slider a practical replacement for the old “wrinkle/detail” idea. It does not try to describe visual wrinkles directly. Instead, it describes the actual production consequence: more small parts versus more large parts.

Why this is useful

The calculator is not meant to give a perfect final bill of materials. It is a fast planning tool.

It helps estimate whether a LEGO model is likely to be small or large, light or heavy, simple or detail-heavy, and whether a client target is realistic before detailed modeling begins.

It is especially useful when only partial information is available, for example:

  • only one model dimension,
  • only a target piece count,
  • only a target weight,
  • client requirements without a complete 3D model,
  • or an early Stud.io file without final optimization.

Formula explanation

The calculator is based on a few connected estimation formulas. The goal is not to produce a perfect final bill of materials, but to create a fast planning estimate from incomplete information.

1. Envelope volume

The envelope volume is the outer bounding-box volume of the model.

Formula:

Length × Width × Height = Envelope Volume

Where:

  • Length, width, and height are measured in centimeters.
  • Envelope volume is measured in cubic centimeters.
  • This is the full outer volume, including empty space inside or around the LEGO structure.

Example:

A model that is 30 × 20 × 10 cm has:

30 × 20 × 10 = 6,000 cm³ envelope volume.

2. Void and occupied LEGO volume

Void is the percentage of the envelope volume that is empty space.

Formula:

Occupied Volume = Envelope Volume × (1 - Void)

Or, if the occupied LEGO volume is known:

Envelope Volume = Occupied Volume / (1 - Void)

Where:

  • Void = 50% means 0.50.
  • Occupied volume means the part of the volume that is actually filled by LEGO structure.
  • Envelope volume means the full external bounding box.

Example:

If the envelope volume is 6,000 cm³ and void is 50%:

Occupied Volume = 6,000 × (1 - 0.50) = 3,000 cm³

This means only half of the outer box is actually occupied by LEGO structure.

3. Detail balance

Detail balance controls how much of the occupied LEGO volume is made from small dense parts versus large hollow structural parts.

A low detail balance means the model is mostly large structural LEGO elements. This usually produces a lighter model with fewer parts.

A high detail balance means the model uses more small decorative or dense parts. This usually increases both part count and weight.

The calculator converts detail balance into a small-part volume share.

Conceptually:

Small Dense Volume Share = Base Small Share + Detail Range × Detail Balance^1.6
Large Structural Volume Share = 1 - Small Dense Volume Share

This does not mean that 50% detail balance equals 50% small parts. The conversion is intentionally nonlinear: 50% detail balance produces a moderate small-part volume share, while 100% represents a very small-part-heavy build.

4. Part count estimate

Part count is estimated from occupied volume and detail balance.

The model is divided into two simplified groups:

  • large structural parts;
  • small dense parts.

Conceptual formula:

Estimated Parts = Occupied Volume × (Small Volume Share / Average Small Part Volume + Large Volume Share / Average Large Part Volume)

Small parts produce more pieces per cubic centimeter. Large parts produce fewer pieces per cubic centimeter.

This is why increasing detail balance increases the estimated part count even if the model dimensions stay the same.

5. Weight estimate

Weight is estimated from occupied LEGO volume, material density, and internal fill.

Formula:

Estimated Weight = Occupied Volume × Weighted Internal Fill × ABS Density

Where:

  • Occupied Volume is the LEGO-occupied space in cm³.
  • Weighted Internal Fill estimates how much of that space is actual plastic.
  • ABS Density is the approximate density of LEGO plastic.
  • The calculator uses about 1.05 g/cm³ as a practical ABS density value.

Large LEGO elements are treated as relatively hollow. Small LEGO elements are treated as denser.

So the weighted internal fill depends on detail balance:

Weighted Internal Fill = Small Volume Share × 65% Small Part Fill + Large Volume Share × Large Part Fill

This is why a model with more small parts becomes heavier even when the outer dimensions remain the same.

6. Large-part internal fill

Large structural LEGO parts are not solid blocks. They contain walls, studs, tubes, holes, and empty space.

The calculator estimates their internal fill based on an effective structural size, but keeps it in a hollow-structure range rather than treating large elements as dense blocks.

Conceptually:

Larger structural elements → about 18–22% internal fill
Smaller structural elements → about 25–30% internal fill

This prevents the model from assuming that every cubic centimeter of occupied LEGO structure is solid plastic.

7. Working backwards from desired part count

If the user enters a desired part count, the calculator works backwards.

It estimates the occupied volume needed to produce that number of parts using the current detail balance.

Then it applies void to calculate the outer envelope volume.

Then it converts that volume into dimensions.

Simplified logic:

Desired Parts → Required Occupied Volume → Envelope Volume → Length / Width / Height

This is useful when the project starts from a piece-count target rather than from dimensions.

8. Working backwards from desired weight

If the user enters a desired weight, the calculator also works backwards.

Formula:

Required Occupied Volume = Desired Weight / (Weighted Internal Fill × ABS Density)

Then:

Envelope Volume = Required Occupied Volume / (1 - Void)

Then the envelope volume is converted into dimensions.

Simplified logic:

Desired Weight → Required Occupied Volume → Envelope Volume → Length / Width / Height

This is useful when the project is limited by shipping weight, material cost, or packaging weight.

9. Dimension solving

When the calculator knows the target envelope volume but not all dimensions, it solves the missing dimensions automatically.

If no dimensions are fixed:

Length = Width = Height = cube root of Envelope Volume

If one dimension is fixed:

The other two dimensions are calculated equally.

Width = Height = square root of (Envelope Volume / Fixed Length)

If two dimensions are fixed:

The third dimension is calculated directly.

Height = Envelope Volume / (Length × Width)

This lets the calculator preserve total volume while adapting the unlocked dimensions.

10. Voxel-based proportion limits

The calculator also prevents unrealistic proportions.

The reference occupied cube is treated as a 6 × 6 × 6 voxel structure.

The voxel size is calculated from the occupied cube before void is added. Then void can increase the outer envelope, but the voxel size itself does not change.

This creates a practical minimum cross-section. A model cannot become infinitely thin just because one dimension is stretched.

In simple terms:

  • The occupied LEGO structure has a minimum meaningful thickness.
  • If one side becomes very long, the other two sides cannot shrink below the voxel-based limit.
  • This prevents unrealistic shapes like a 200 × 1 × 1 cm model.

11. Lock system as formula constraints

The lock system turns the calculator into a constraint solver.

A locked value is treated as fixed in the formulas.

An unlocked value is allowed to change.

When the user moves any slider, that value becomes the active driver. The calculator then recalculates all unlocked dependent values while keeping locked values unchanged.

This means the same formulas can work in different directions:

  • dimensions → parts and weight;
  • parts → volume and dimensions;
  • weight → volume and dimensions;
  • fixed dimensions + target weight → detail balance, void, or unlocked dimensions;
  • fixed length + target parts → width and height.

The calculator is therefore not a one-way estimator. It is a flexible planning model for exploring LEGO model size, weight, part count, void, and detail density together.
