Usage Scenarios

How the linked LEGO estimator behaves when different values are moved or locked.

Full documentation loaded

Core idea

This estimator treats a LEGO model as a simplified structure made from fixed 8 mm voxel slots. One voxel is approximately the volume of a small LEGO detail slot: 0.8 × 0.8 × 0.8 cm.

The calculator is not trying to predict the exact final bill of materials. It estimates whether a model of a given size, part count, weight, void, and detail level is physically plausible as a LEGO build.

The most important hidden value is q: average bearing coverage. It describes how many occupied voxels one large structural/bearing element can effectively cover. Larger q means fewer large parts are needed to span the model.

How the locks work

Every visible value can be locked. Locked values become fixed anchors. Unlocked values are allowed to adapt when another value is changed.

  • If a dimension is locked, the model must keep that physical size.
  • If desired parts are locked, the estimator changes hidden q, void, balance, or other unlocked values to reach that part count.
  • If desired weight is locked, the estimator adjusts the reachable structure so the plastic volume matches the target weight as closely as possible.
  • At least two visible values must remain unlocked so the model has room to adapt.

Moving a slider makes it the active driver. The calculator then solves the other unlocked values from the fixed 8 mm voxel model.

the fixed 8 mm voxel model.

Scenario 1: dimensions are known

When length, width, and height are locked, the envelope volume is fixed. The estimator calculates how many 8 mm voxel slots fit inside that bounding box.

Envelope volume = Length × Width × Height
Envelope voxels = Envelope volume / 0.512

Void then decides how many of those voxels are actually occupied by LEGO structure.

This is useful when a Stud.io model already exists, or when the client gives a required physical size.

Scenario 2: aircraft, cranes, signs, and other open models

Large, thin, or spread-out models can have a high external void because the bounding box contains a lot of empty space. Aircraft are a typical example: wings and tail increase the envelope, but the actual LEGO structure occupies only part of that space.

In this version, high void should not be solved by artificially increasing small-detail density. Instead, low part count is mainly handled by increasing hidden bearing coverage q.

That means an aircraft can stay low-detail, use large structural spans, and still remain plausible without forcing the model toward 99% void.

Scenario 3: compact models

Compact objects usually have lower void and smaller q. Their volume is more densely occupied, and the same part count can produce a smaller model.

For a compact model, the estimator expects fewer empty voxel slots and a more direct relationship between physical volume, weight, and part count.

If the calculator says the closest reachable estimate is outside the target, it usually means the locked part count or weight is too strict for the selected dimensions and void.

Scenario 4: changing external void

External void describes empty space inside the model’s bounding box. It is a geometric value, not the hollow space inside individual LEGO bricks.

The normal working range is now:

Void range = 5% to 95%

Low void means the envelope is mostly filled with LEGO structure. High void means the silhouette is spread out, open, thin, or sparse.

When void increases, occupied voxels decrease. That usually lowers weight and may require larger q if the desired part count is locked.

Scenario 5: changing small / large detail balance

Detail balance controls how many occupied voxels are reserved for non-bearing small details.

Small details do not work as bearings. They add detail, part count, and weight, but they do not help span large distances.

  • 0% balance: almost all occupied voxels are large structural/bearing voxels.
  • 50% balance: moderate detail density.
  • 100% balance: many small details, but the model still keeps a large structural core.

This prevents the calculator from solving low part count by incorrectly turning the whole model into small details.

Scenario 6: desired parts are fixed

If desired parts are locked, the estimator calculates the hidden q value needed to make the model possible.

If q stays within its allowed range, the model is reachable. If q would need to be too small or too large, the target is outside the current feasible range.

For example, a large aircraft with only a few hundred parts needs a high q value, meaning the large parts are treated as long structural spans rather than many small separate pieces.

Scenario 7: desired weight is fixed

If desired weight is locked, the estimator compares the target weight against the plastic volume implied by occupied voxels, small-detail fill, large-structure fill, and ABS density.

Weight is not calculated directly from the number of parts. It is calculated from estimated plastic volume.

This allows two models with similar part counts to have different weights if one uses denser small details and the other uses larger hollow structural spans.

What the estimator is good for

The calculator is best used during early planning, before a final optimized Stud.io model exists.

  • It can test whether a target part count is plausible for a given size.
  • It can estimate approximate weight from dimensions and structure type.
  • It can compare compact objects, vehicles, buildings, aircraft, signs, and open structures.
  • It can show when a client request is outside a reasonable LEGO build range.

The result should be read as a planning estimate, not as a final production BOM.

Formula Notes

Formula explanation and calculation logic used by the LEGO estimator.

Full formula notes loaded

1. Fixed voxel size

The estimator uses a fixed voxel size of 8 mm. This matches the basic LEGO module width and gives the model a stable physical unit.

voxelSide = 0.8 cm
voxelVolume = 0.8 × 0.8 × 0.8 = 0.512 cm³

A small model does not force the voxel size to stay large. Instead, it simply contains fewer voxels. For example, a 2.4 × 2.4 × 2.4 cm model contains 3 × 3 × 3 = 27 envelope voxels.

2. Envelope volume and envelope voxels

The envelope volume is the outer bounding box of the model.

Venv = L × W × H

The number of envelope voxels is calculated from the physical volume:

Nenv = Venv / voxelVolume

Since voxelVolume is fixed at 0.512 cm³, larger models naturally contain more voxel slots.

3. External void

External void is the percentage of the envelope that is not occupied by LEGO structure.

void range = 5% to 95%
Nocc = Nenv × (1 - void)

Here, Nocc is the number of occupied LEGO structure voxels. Void is geometric empty space in the bounding box, not the hollow space inside individual LEGO parts.

4. Detail balance

Detail balance controls how many occupied voxels are assigned to small non-bearing details.

The current model reserves up to 25% of occupied voxels for small details:

smallVoxelFraction = 0.25 × balance
Nsmall = Nocc × smallVoxelFraction
Nlarge = Nocc - Nsmall

Examples:

  • balance = 0% → all occupied voxels are large structural voxels.
  • balance = 50% → 12.5% of occupied voxels are small-detail voxels.
  • balance = 100% → 25% of occupied voxels are small-detail voxels.

5. Small details

Small details are treated as non-bearing elements. They add part count and weight, but they do not span or connect multiple structural voxel zones.

Psmall = Nsmall

This keeps small details from incorrectly solving large, low-part-count models. A model such as an aircraft should reduce part count mostly by increasing bearing coverage, not by changing small-detail behavior.

6. Hidden bearing coverage q

The large structural part count is controlled by hidden average bearing coverage q.

q means: how many occupied large voxels are effectively covered by one large structural/bearing element.

Plarge = Nlarge / q

The current bearing limit is based on the idea of a maximum 1 × 1 × 16 voxel structural span:

qMin = 1
qMax = min(16, longest side in voxels)

Since one voxel is 8 mm, 16 voxels correspond to 128 mm. This keeps q connected to a plausible large LEGO structural span while still remaining an estimate.

7. Total estimated parts

Total part count is the sum of non-bearing small details and large structural parts:

P = Psmall + Plarge
P = Nsmall + Nlarge / q

If desired parts are locked, q can be solved directly:

qRequired = Nlarge / (Ptarget - Nsmall)

The target is feasible when:

Ptarget > Nsmall and qMin ≤ qRequired ≤ qMax

8. Weight model

Weight is calculated from estimated plastic volume, not directly from part count.

Weight = Vplastic × ABSdensity

Plastic volume is estimated from small-detail fill and large-structure fill:

Vplastic = voxelVolume × (Nsmall × smallFill + Nlarge × largeFill(q))
Weight = voxelVolume × (Nsmall × smallFill + Nlarge × largeFill(q)) × ABSdensity

Default assumptions:

  • ABSdensity ≈ 1.05 g/cm³
  • smallFill ≈ 0.65
  • largeFill(q) decreases as q grows, because larger structural spans are treated as more efficient/hollow.

9. Large-structure fill

Large LEGO structural elements are not solid blocks. They are treated as hollow or semi-hollow structural coverage.

A simple fill curve can be:

largeFill(q) = 0.18 + 0.12 × (1 / q)^0.6

This gives approximately:

  • q = 1 → denser large structure, about 30% fill.
  • q = 16 → more efficient large span, about 20% fill.

10. Dimension limits

The smallest useful model side is 2.4 cm, corresponding to 3 voxels.

minimum side = 3 × 0.8 cm = 2.4 cm

For proportion control, extremely thin structures can be limited by requiring meaningful cross-section. A practical structural rule is that long, stick-like forms should not become thinner than a stable voxel cross-section.

The estimator still remains an approximation: it should say whether a model can plausibly exist, not whether every real LEGO connection is already solved.

11. Why this model is faster

The previous solver searched through many combinations of void, balance, and bearing size. This version avoids that by solving q directly when the desired part count is known.

qRequired = Nlarge / (Ptarget - Nsmall)

This turns the hardest part of the calculation into a direct equation instead of a heavy search.

12. Interpretation

The estimator should be read as a feasibility and planning tool.

  • High void means open bounding-box geometry.
  • High q means large structural spans and fewer large parts.
  • High balance means more non-bearing small-detail voxels.
  • Higher weight means more plastic volume, not necessarily more parts.

A valid estimate means the model is plausible under the selected assumptions. It does not mean the final LEGO model is already optimized or mechanically verified.
