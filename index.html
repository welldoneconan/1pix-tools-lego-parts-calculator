Usage Scenarios

How to use the estimator when only part of the project information is known. The calculator behaves like a constraint system: locked values stay fixed and unlocked values adapt.

1. Starting from model dimensions

When length, width, and height are known from a Stud.io file or client requirements, the calculator uses them as the envelope size. It applies void to estimate occupied LEGO volume, then calculates part count and weight.

This is the most direct scenario: dimensions → volume → parts and weight.

2. One dimension is fixed

If one side is locked, for example length, the calculator keeps that value fixed. When the user changes desired weight or desired part count, width and height adapt together.

This is useful when the client has a strict maximum length but the other dimensions can still change.

3. Two dimensions are fixed

If two sides are locked, the third side becomes the main adjustable dimension. For example, if length and width are fixed, height changes to satisfy the target volume, part count, or weight.

This is useful for models that must fit a shelf, baseplate, package insert, or display footprint.

4. All dimensions are fixed

If all three dimensions are locked, the physical bounding box is fixed. The calculator then updates only the remaining unlocked values such as estimated parts, weight, void, or detail balance.

This is useful when the final model size is already decided and the estimator is used to understand production impact.

5. Starting from desired part count

When only desired part count is known, the calculator works backwards. It estimates how much occupied LEGO volume is needed for that number of parts, using current void and detail balance assumptions.

Then it converts the result into model dimensions. If no dimensions are locked, the result is cube-like. If dimensions are locked, the remaining dimensions adapt.

6. Starting from desired weight

When desired weight is the main target, the calculator estimates the occupied LEGO volume needed to reach that weight. It uses detail balance to estimate how dense the LEGO structure is.

This is useful for shipping constraints, packaging limits, material cost planning, or early feasibility checks.

7. Changing void

Void describes how much empty space exists inside the outer model envelope. Higher void means the same LEGO structure occupies a larger external bounding box.

If void is locked, it becomes a fixed design assumption. If void is unlocked, it can adapt when dimensions, part count, or weight are locked.

8. Changing detail balance

Detail balance replaces the earlier “wrinkle/detail” idea. It does not describe visual wrinkles directly. It describes the production consequence: the balance between small dense parts and large hollow structural parts.

Higher detail balance usually increases both part count and weight for the same occupied LEGO volume.

Formula Notes

The formulas are simplified planning estimates. They are designed for fast feasibility checks, not for a final bill of materials.

Envelope volume

The envelope volume is the outer bounding-box volume of the LEGO model.

Envelope Volume = Length × Width × Height

All dimensions are measured in centimeters, so the result is cubic centimeters.

Void and occupied LEGO volume

Void is the empty share of the envelope volume. Occupied LEGO volume is the part of the bounding box that is actually filled by LEGO structure.

Occupied Volume = Envelope Volume × (1 - Void)
Envelope Volume = Occupied Volume / (1 - Void)

Detail balance

Detail balance controls how much occupied LEGO volume is treated as small dense detail parts versus large hollow structural parts.

Small Dense Volume Share = 0.05 + 0.30 × Detail Balance
Large Structural Volume Share = 1 - Small Dense Volume Share

A 50% balance does not mean half the pieces are small. It means the model sits halfway between a mostly structural build and a highly detailed build.

Part count estimate

The model uses two simplified groups: small dense parts and large structural parts.

Estimated Parts = Occupied Volume × (Small Share / Small Part Volume + Large Share / Large Part Volume)

Small parts generate more pieces per cubic centimeter. Large parts cover more volume with fewer pieces.

Weight estimate

Weight is based on occupied volume, ABS density, and effective internal fill.

Estimated Weight = Occupied Volume × Weighted Internal Fill × ABS Density
Weighted Internal Fill = Small Share × Small Fill + Large Share × Large Fill

The calculator uses 1.05 g/cm³ as a practical ABS density value. Small parts are treated as denser, while large parts are treated as more hollow.

Large-part internal fill

Large structural LEGO elements are not solid blocks. They include walls, studs, tubes, holes, and internal empty space.

Large Fill = 0.20 + 0.80 × (8 / Effective Size)^0.85

This prevents the calculator from treating every occupied cubic centimeter as solid plastic.

Working backwards from part count

When desired part count is the driver, the calculator searches for the occupied volume that produces that count under the current balance assumption.

Desired Parts → Required Occupied Volume → Envelope Volume → Dimensions

Working backwards from weight

When desired weight is the driver, the calculator estimates required occupied volume from density and fill, then converts it into envelope size.

Required Occupied Volume = Desired Weight / (Weighted Fill × ABS Density)

Solving missing dimensions

If the target envelope volume is known but some dimensions are unlocked, the calculator solves the missing dimensions automatically.

No fixed dimensions: L = W = H = ∛Envelope Volume
One fixed dimension: W = H = √(Envelope Volume / Fixed Length)
Two fixed dimensions: H = Envelope Volume / (Length × Width)

Voxel-based proportion limits

The occupied cube is treated as a 6 × 6 × 6 voxel structure. Voxel size is calculated before void is added, then remains fixed after void changes the envelope.

This creates a practical minimum cross-section and prevents unrealistic shapes such as extremely long and thin models.

Lock system as formula constraints

A locked value is treated as fixed. An unlocked value is allowed to adapt. When the user moves any slider, that value becomes the active driver and all unlocked dependent values are recalculated.

This lets the same formulas work in different directions: dimensions to parts, parts to dimensions, weight to volume, or fixed dimensions to balance and void.
